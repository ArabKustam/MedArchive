## 📌 О проекте

**MedArchive** — это высокотехнологичный программный комплекс для автоматического сбора, парсинга, нормализации и анализа прайс-листов медицинских клиник-партнёров.

### 💡 Какую проблему решает проект?
Клиники присылают свои прайс-листы в самых разных форматах: PDF-сканы, многовкладочные таблицы Excel (`.xlsx`, `.xls`), документы Word (`.docx`), а также архивы со смешанным содержимым. Названия одних и тех же процедур отличаются у каждой клиники. 

**MedArchive** полностью автоматизирует этот процесс:
1. Принимает любые форматы документов и ZIP-архивы.
2. Распознает таблицы и текст (включая не распознанные сканы через OCR и нейросети).
3. Приводит названия услуг к единому эталонному справочнику с помощью алгоритмов нечёткого поиска (`Fuzzy Matching`) и ИИ (`LLM`).
4. Предоставляет операторам умную очередь верификации со спорными позициями.
5. Предоставляет пользователям удобный поиск с фильтрацией и сравнением цен.

## 🔄 Схема работы парсера файлов и нормализации (Step-by-Step)

Ниже подробно визуализирован полный путь файла: от момента загрузки ZIP-архива пользователем до записи очищенных услуг в базу данных.

```
┌──────────────────────────────────────────────────────────────────────────────────────────┐
│ 1. ЗАГРУЗКА И РАСПАКОВКА                                                                 │
│   [Пользователь] ──► Upload ZIP/PDF/XLSX ──► Временное хранилище /storage/uploads        │
│                                              │                                           │
│                                              ▼ (Распаковка zipfile)                      │
│                                    Список файлов по клиникам                             │
└──────────────────────────────────────────────┬───────────────────────────────────────────┘
                                               │
                                               ▼
┌──────────────────────────────────────────────────────────────────────────────────────────┐
│ 2. ОПРЕДЕЛЕНИЕ ФОРМАТА И ИЗВЛЕЧЕНИЕ ТЕКСТА / ТАБЛИЦ (Парсинг)                             │
│                                                                                          │
│   ├── Excel (.xlsx, .xls)  ──► python-calamine (Rust-engine) / openpyxl                  │
│   ├── PDF (.pdf)          ──► pdfplumber (извлечение табличной структуры)                │
│   ├── Word (.docx)         ──► python-docx (парсинг параграфов и таблиц)                 │
│   └── Сканы / Фото (OCR)   ──► Google Cloud Vision API / pytesseract + pdf2image          │
│   └── Сложные неформаты    ──► Groq API (Модель LLaMA 3.3 70B / 3.1 8B)                    │
└──────────────────────────────────────────────┬───────────────────────────────────────────┘
                                               │ Сырые строки: [Название, Цена, Код]
                                               ▼
┌──────────────────────────────────────────────────────────────────────────────────────────┐
│ 3. ОЧИСТКА И ФИЛЬТРАЦИЯ СЫРЫХ ДАННЫХ                                                     │
│   • Удаление шапок, подвалов, контактов клиники и лицензий                               │
│   • Очистка цен (удаление "руб.", "BYN", мусорных символов, приведение к float)          │
│   • Фильтрация пустых или невалидных строк                                               │
└──────────────────────────────────────────────┬───────────────────────────────────────────┘
                                               │ Очищенная позиция
                                               ▼
┌──────────────────────────────────────────────────────────────────────────────────────────┐
│ 4. НОРМАЛИЗАЦИЯ И МАТЕМАТИЧЕСКОЕ СОПОСТАВЛЕНИЕ (Fuzzy Matching)                          │
│   Сравнение сырой строки с эталонным справочником услуг (Service) через RapidFuzz        │
│                                                                                          │
│   • Точное совпадение (100%)              ──► Confidence Score = 1.0  ──► Status: AUTO   │
│   • Высокая схожесть (Token Set Ratio >= 85%)──► Confidence Score >= 0.85 ──► Status: AUTO   │
│   • Частичное совпадение (60% - 84%)      ──► Confidence Score 0.60-0.84──► Status: REVIEW │
│   • Совпадение не найдено (< 60%)         ──► Confidence Score < 0.60 ──► Status: UNMATCHED│
└──────────────────────────────────────────────┬───────────────────────────────────────────┘
                                               │
                                               ▼
┌──────────────────────────────────────────────────────────────────────────────────────────┐
│ 5. СОХРАНЕНИЕ В БАЗУ ДАННЫХ И МАРШРУТИЗАЦИЯ                                               │
│   ├── Status == AUTO     ──► Запись в PriceItem (Показывается в общем поиске)            │
│   └── Status == REVIEW   ──► Попадает в панель модератора (AdminVerifyPage)              │
└──────────────────────────────────────────────────────────────────────────────────────────┘
```

## 🗄 Структура Базы Данных (ER-Диаграмма)

Система использует реляционную базу данных SQLite (`medarchive.db`) с поддержкой режима WAL (Write-Ahead Logging) для параллельного чтения и записи.

### Визуальная схема связей таблиц (ER-Diagram):

```
┌──────────────────────────┐       1:N       ┌──────────────────────────┐
│         partners         │─────────────────│      price_documents     │
├──────────────────────────┤                 ├──────────────────────────┤
│ PK partner_id (INT)      │                 │ PK doc_id (INT)          │
│    name (VARCHAR)        │                 │ FK partner_id (INT)      │
│    city (VARCHAR)        │                 │    filename (VARCHAR)    │
│    address (TEXT)        │                 │    status (queued/parsed)│
│    created_at (DATETIME) │                 │    price_date (DATETIME) │
└──────────────────────────┘                 └────────────┬─────────────┘
                                                          │ 1:N
                                                          ▼
┌──────────────────────────┐       N:1       ┌──────────────────────────┐
│         services         │◄────────────────│        price_items       │
├──────────────────────────┤                 ├──────────────────────────┤
│ PK service_id (INT)      │                 │ PK item_id (INT)         │
│    code (VARCHAR)        │                 │ FK doc_id (INT)          │
│    name (VARCHAR)        │                 │ FK service_id (INT)      │
│    category (VARCHAR)    │                 │    raw_service_name (TXT) │
│    created_at (DATETIME) │                 │    price (NUMERIC)       │
└──────────────────────────┘                 │    confidence_score (FLT)│
             ▲                               │    status (auto/review)  │
             │ 1:N                           └────────────┬─────────────┘
┌────────────┴─────────────┐                              │ 1:N
│      price_history       │                              ▼
├──────────────────────────┤                 ┌──────────────────────────┐
│ PK history_id (INT)      │                 │      price_history       │
│ FK service_id (INT)      │                 │ (Фиксация изменения цен) │
│ FK item_id (INT)         │                 └──────────────────────────┘
│    price (NUMERIC)       │
│    recorded_at (DATETIME)│
└──────────────────────────┘
```

### Детальное описание таблиц БД:
1. **`partners` (Клиники-партнёры)** — хранит информацию о медицинских центрах (название, город, контактные данные).
2. **`price_documents` (Прайс-листы)** — реестр загруженных файлов с фиксацией статуса обработки (`queued`, `processing`, `completed`, `error`), даты прайса и пути к оригиналу.
3. **`services` (Справочник услуг)** — эталонный медицинский каталог (код услуги, нормализованное наименование, категория).
4. **`price_items` (Распарсенные позиции)** — конкретные услуги из прайсов со связью с эталоном, сырым названием, выделенной ценой и процентом уверенности сопоставления (`confidence_score`).
5. **`price_history` (История цен)** — фиксирует изменение стоимости процедур во времени для аналитики и графиков на дашборде.

## 🏗 Технологический стек и Архитектура

### 🎨 Frontend (Клиентская часть)
* **React 19** — библиотек интерфейсов с поддержкой современных хуков и серверных компонентов.
* **TypeScript 5.8** — обеспечивает 100% строгую типизацию данных.
* **Feature-Sliced Design (FSD)** — архитектура раскладки кода по изолированным слоям (`pages`, `widgets`, `features`, `entities`, `shared`).
* **Vite 8** — быстрый сборщик и dev-сервер.
* **TanStack Router** — типобезопасный файловый роутинг.
* **TanStack React Query v5** — кэширование и фоновая синхронизация API-запросов.
* **Tailwind CSS v4 & Radix UI** — адаптивный дизайн и доступные UI-компоненты.
* **Recharts** — интерактивные графики и визуализация аналитики.

### ⚙️ Backend (Серверная часть)
* **Python 3.11+** — основной язык разработки.
* **FastAPI** — асинхронный REST API фреймворк.
* **Uvicorn** — боевой ASGI-сервер.
* **SQLAlchemy 2.0 & SQLite** — ORM для работы с базой данных `medarchive.db` в режиме WAL.
* **Pydantic v2** — строгая валидация входящих и исходящих структур данных.

## 🚀 Пошаговая инструкция по запуску

### Способ 1. Автоматический запуск в один клик (Windows)
В корневой папке репозитория находятся готовые батники:
1. Кликните дважды на **`start.bat`** (или `run.bat`). Скрипт сам запустит бэкенд и фронтенд.
2. Для остановки сервисов запустите **`stop.bat`**.

---

### Способ 2. Ручной запуск для разработки

#### Шаг 1. Запуск серверной части (Backend)
```bash
# 1. Перейдите в каталог бэкенда
cd backend

# 2. Создайте и активируйте виртуальное окружение Python
python -m venv .venv

# Для Windows (PowerShell / CMD):
.venv\Scripts\activate
# Для Linux / macOS:
source .venv/bin/activate

# 3. Установите необходимые зависимости
pip install -r requirements.txt

# 4. Запустите сервер Uvicorn
python -m uvicorn app.main:app --reload --host 0.0.0.0 --port 8765
```
📍 **Адреса бэкенда:**
* REST API: `http://localhost:8765`
* Интерактивная документация Swagger: `http://localhost:8765/docs`

#### Шаг 2. Запуск клиентской части (Frontend)
```bash
# 1. Вернитесь в корень проекта
cd ..

# 2. Установите зависимости Node.js через bun или npm
npm install
# или
bun install

# 3. Запустите dev-сервер Vite
npm run dev
# или
bun dev
```
📍 **Адрес фронтенда:**
* Веб-приложение: `http://localhost:5173` (или `http://localhost:8080`)

## 🖥 Обзор страниц и возможностей приложения

| Экран / Модуль | Описание и Функционал |
|---|---|
| 🔍 **Поиск услуг (`SearchPage`)** | Моментальный поиск по всему архиву прайсов. Поддерживает сортировку по цене, релевантности и точности совпадения, имеет пагинацию и возможность выгрузки в CSV/XLSX. |
| 📊 **Дашборд (`AdminDashboardPage`)** | Главная панель аналитики. Выводит ключевые метрики (всего услуг, обработано клиник, процент верификации) и интерактивные графики цен на базе Recharts. |

| ⚖️ **Очередь верификации (`AdminVerifyPage`)** | Рабочее место модератора. Показывает позиции со статусом `review` или нетипичными ценами для быстрого утверждения или ручной корректировки. |
| 🏥 **Клиники-партнёры (`PartnersListPage`)** | Реестр всех клиник с детальной информацией по каждой (город, количество прайсов, история обновлений). |
| 📋 **Каталог услуг (`ServicesListPage`)** | Эталонный справочник медицинских процедур и исследований. |

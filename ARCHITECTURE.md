# MedPartners — Архитектура проекта

> Система автоматической обработки архива прайс-листов клиник-партнёров и формирование базы услуг и цен.

---

## 1. Общая архитектура

Проект построен на **клиент-серверной архитектуре** с чётким разделением:

```
┌─────────────────────────────┐       REST API (JSON)       ┌──────────────────────────────┐
│        FRONTEND             │  ◄───────────────────────►   │          BACKEND             │
│   React 19 + TanStack       │       http://localhost:8765  │   FastAPI + SQLAlchemy        │
│   Vite (порт 5173)          │                              │   Uvicorn (порт 8765)         │
└─────────────────────────────┘                              └──────────┬───────────────────┘
                                                                        │
                                                              ┌─────────▼─────────┐
                                                              │   SQLite (файл)    │
                                                              │   medarchive.db    │
                                                              └───────────────────┘
```

### 2.3. Парсинг документов

| Библиотека | Назначение |
|---|---|
| **pdfplumber** ≥0.11 | Извлечение таблиц и текста из PDF-файлов с текстовым слоем |
| **python-docx** ≥1.1 | Чтение таблиц из DOCX (Word) документов |
| **python-calamine** ≥0.2 | Быстрое чтение XLSX/XLS через Rust-binding (быстрее openpyxl в 10-50x) |
| **openpyxl** ≥3.1 | Резервная библиотека для Excel |
| **pandas** ≥2.2 | Доп. обработка табличных данных при необходимости |

**Почему такой стек:**
- **python-calamine** — написан на Rust, читает большие Excel-файлы за миллисекунды, поддерживает `.xls` и `.xlsx`.
- **pdfplumber** — специализируется на извлечении таблиц из PDF, в отличие от PyPDF2 который работает только с текстом.

### 2.4. OCR (оптическое распознавание)

| Библиотека | Назначение |
|---|---|
| **Google Cloud Vision API** ≥3.7 | Основной OCR — распознавание текста на сканированных документах |
| **pdf2image** ≥1.17 | Конвертация страниц PDF в изображения (JPEG) для OCR |
| **Pillow** ≥10.0 | Обработка изображений (формат, поворот, подготовка к OCR) |
| **pytesseract** ≥0.3.10 | Резервный OCR (Tesseract) если Google Vision недоступен |

**Почему Google Vision, а не только Tesseract:**
- Google Cloud Vision API даёт **значительно лучшее качество** на кириллических документах, особенно на сканах с шумом.
- Tesseract оставлен как fallback для автономной работы без интернета.
- Цепочка: PDF → `pdf2image` (растеризация 200dpi) → JPEG → Google Vision API → текст → парсинг.

### 2.5. Нормализация и сопоставление

| Библиотека | Назначение |
|---|---|
| **RapidFuzz** ≥3.6 | Нечёткое сопоставление строк (fuzzy matching) для сопоставления услуг |
| **sentence-transformers** (опционально) | Семантическое сопоставление через ML-эмбеддинги |

**Как работает сопоставление (4 уровня):**
1. **Точное совпадение** — строка в прайсе идентична каталогу → `auto` (100%)
2. **Fuzzy matching (RapidFuzz)** — `token_set_ratio` ≥ 85% → `auto`, ≥ 60% → `review`
3. **Семантическое сопоставление** — модель `paraphrase-multilingual-MiniLM-L12-v2` вычисляет cosine similarity эмбеддингов
4. **Unmatched** — ничего не подошло, требуется ручная верификация

## 4. Модель данных (БД)

```
┌─────────────┐       ┌──────────────────┐       ┌──────────────┐
│  partners   │ 1───N │ price_documents  │ 1───N │ price_items   │
│─────────────│       │──────────────────│       │──────────────│
│ partner_id  │       │ doc_id           │       │ item_id      │
│ name        │       │ partner_id (FK)  │       │ doc_id (FK)  │
│ city        │       │ filename         │       │ partner_id   │
│ status      │       │ file_type        │       │ service_id   │
│ created_at  │       │ price_date       │       │ service_name │
└─────────────┘       │ status           │       │ price_res    │
                      │ items_total      │       │ price_nonres │
                      │ items_matched    │       │ match_score  │
                      └──────────────────┘       │ match_status │
                                                 └──────┬───────┘
                                                        │
┌──────────────┐                             ┌──────────▼───────┐
│  services    │                             │ price_history    │
│──────────────│                             │──────────────────│
│ service_id   │  ◄──────────────────────    │ history_id       │
│ service_name │                             │ partner_id       │
│ category     │                             │ service_id       │
│ synonyms[]   │                             │ price_resident   │
│ is_active    │                             │ effective_date   │
└──────────────┘                             └──────────────────┘
```

**5 таблиц:**
- `partners` — клиники-партнёры (автоматически создаются из имён файлов)
- `services` — каталог-справочник медицинских услуг
- `price_documents` — загруженные документы (из ZIP-архива)
- `price_items` — извлечённые позиции прайса (услуга + цена + статус сопоставления)
- `price_history` — журнал изменений цен (append-only, для отслеживания динамики)

## 5. Пайплайн обработки документа

```
ZIP-архив
    │
    ▼
1. РАСПАКОВКА (iter_archive)
    │  — Декодирование кириллических имён файлов (CP866/CP1251)
    │  — Фильтрация по расширениям: PDF, DOCX, XLSX, XLS, JPG, PNG
    ▼
2. ПАРСИНГ (parse_file)
    │  — XLSX/XLS: python-calamine → поиск заголовка → маппинг столбцов
    │  — DOCX: python-docx → таблицы → поиск заголовка → маппинг
    │  — PDF (текстовый): pdfplumber → извлечение таблиц
    │  — PDF (скан): pdf2image → Google Vision API / Tesseract → текст → парсинг
    │  — Результат: список ExtractedRow(name, price_res, price_nonres, code)
    ▼
3. ВАЛИДАЦИЯ (validation.py)
    │  — Проверка: цена > 0, имя не пустое
    │  — Конвертация валют (USD/EUR/RUB → KZT)
    │  — Детекция аномалий: отклонение цены > 50% от предыдущей
    │  — Предупреждения: нерезидент < резидент, дата в будущем
    ▼
4. НОРМАЛИЗАЦИЯ (normalize.py)
    │  — Exact match → auto (100%)
    │  — RapidFuzz token_set_ratio ≥ 85% → auto
    │  — Sentence-Transformers cosine sim → auto/review
    │  — Ниже 60% → unmatched (в очередь верификации)
    ▼
5. СОХРАНЕНИЕ (pipeline.py)
    │  — Создание Partner (если нет)
    │  — Создание PriceDocument
    │  — Создание PriceItem для каждой позиции
    │  — Деактивация старых цен (is_active = False)
    │  — Запись в PriceHistory
    ▼
ГОТОВО → данные доступны через API и UI
```

## 7. Ключевые алгоритмы

### 7.1. Интеллектуальное определение столбцов (parse_xlsx / parse_docx)

Документы разных клиник имеют **разное количество столбцов и разный порядок**:
- Клиника 6: 9 столбцов (№, Код, Наименование, Ед.изм., Кол-во, Цена без НДС, Цена резидент, Цена СНГ, Цена дальнее зарубежье)
- Клиника 8: 5 столбцов (№, Наименование, Код, Ед.изм., Цена)
- Клиника 1: 3 столбца (Код, Наименование, Стоимость)

**Алгоритм `_detect_columns()`:**
1. Сканирует первые 30 строк
2. Ищет строку, где есть ключевые слова «наименование» + «цена/стоимость»
3. Определяет индексы: `name_idx`, `code_idx`, `price_res_idx`, `price_nonres_idx`
4. Все последующие строки парсятся **по конкретным индексам**, а не вслепую

### 7.2. Фильтрация мусорных строк

- **Merged cells** (объединённые ячейки): все ячейки строки содержат один текст → пропуск
- **Секционные заголовки**: «ПРИЕМ ВРАЧА», «ГЕМАТОЛОГИЯ», «Раздел 1. Консультации» → пропуск
- **Шапка документа**: «Приказ», «Утверждаю», «Председатель правления» → пропуск
- **Защита от аномальных цен**: значения > 10 000 000 тг отбрасываются

### 7.3. Декодирование имён файлов в ZIP

ZIP-архивы из Windows содержат кириллические имена в кодировке **CP866** или **CP1251**, которые Python читает как CP437 (мусор). Алгоритм:
1. Попробовать декодировать raw bytes как UTF-8
2. Если не UTF-8 — попробовать CP866, CP1251 с подсчётом кириллических символов
3. Выбирается вариант с максимальным количеством кириллицы
